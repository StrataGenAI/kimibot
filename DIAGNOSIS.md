# KimiBot Core Engine Diagnosis

**Date**: 2026-04-21  
**Status**: All issues identified, fixes in progress

---

## Issue 1 — No Ingestion Systemd Service

`run_ingestion_loop()` exists in `ingestion/recorder.py` but is never started automatically. There is no corresponding systemd unit, so the function must be invoked manually via `python main.py ingest`.

**Consequence**: `data/market_snapshots.csv` is the static fixture written at setup time. The health endpoint reads its mtime to compute `data_freshness_seconds` — which currently reports ~41,000+ seconds stale because the file has never been updated by live ingestion.

**Fix**: Create `/etc/systemd/system/kimibot-ingest.service` and add a status JSON write to `run_ingestion_loop()`.

---

## Issue 2 — Synthetic Data (m1–m20 CSV Fixtures)

All market data is synthetic. `data/market_metadata.csv` contains markets with IDs m1–m20 generated by `scripts/setup_synthetic_data.py`. These are not real Limitless markets; they exist solely to give the train/backtest pipeline something to work with during development.

**Implication**: Any edge, PnL, or signal shown in the dashboard is derived from synthetic data. No real money is at risk, but the numbers are not production-valid until live ingestion is wired up and real market data populates the CSVs.

---

## Issue 3 — Test Window / Resolution Time Mismatch (0 Settlements)

The backtest walk-forward engine defines the test window as:

```
test_end = 2026-01-04T23:45:00
```

Markets m17–m20 have `resolution_time = 2026-01-05T00:00:00`.

The event replay loop in `backtest/engine.py` only triggers settlement when `event.timestamp >= meta["resolution_time"]`. Because the replay never advances past 23:45 on Jan 4, resolution events for m17–m20 are never emitted.

**Consequence**:
- `trade_count = 0` in `data/baseline/metrics.json`
- `realized_pnl = 0.0`
- All metrics (Sharpe, win rate, avg edge) are undefined or zero

**Fix**: After the main replay loop, force-settle all open positions for markets whose `outcome_yes` is known in `market_metadata.csv`.

## Issue 4 — Class Imbalance Causing Phantom BUY NO Signals

Training labels m1–m14:
- YES outcomes: 3 (m2, m8, m12)
- NO outcomes: 11 (all others)
- **79% NO** — severe class imbalance

`LogisticRegressionModel.fit()` uses unweighted gradient descent. The model learns to predict near-zero YES probability for every market.

`predict_raw()` in `predictor.py` clips output to `[0.20, 0.80]`. With raw predictions near 0, all clips land at 0.20.

After Platt scaling (calibration), the output remains near 0.20. With market prices around 0.45:

```
edge = p_model_calibrated(0.20) - p_market(0.45) ≈ −0.25
```

Every signal becomes BUY NO with a phantom 25–36pp edge. These edges are not real; they are an artifact of class imbalance driving the model to a degenerate state.

**Fix**: Apply balanced class weighting in `LogisticRegressionModel.fit()` and relax clip bounds from `[0.20, 0.80]` to `[0.05, 0.95]`.

---

## Issue 5 — Insufficient Calibration Data

The sigmoid calibrator (Platt scaling) is fit on a single calibration market (m15). Platt scaling requires a reasonable sample of calibration examples to find stable sigmoid parameters `a` and `b`. With n=1 market, the calibrator is essentially fitting two parameters to a single data point.

**Consequence**: Calibrated probabilities are unreliable. The calibrator may be inverting or heavily distorting model outputs.

**Note**: The held-out calibration split is architecturally correct — calibration markets are separate from training markets (m1–m14), so there is no label leakage. The issue is purely the sample size.

**Fix**: Raise `min_calibration_markets` from 1 to 3, and generate synthetic data with a more balanced train/calibration/test split in future fixture updates.

---

## What Is NOT a Problem

- **No look-ahead bias**: Feature computation in `features/builder.py` uses only data available at `snapshot_time`. The rolling windows (volatility, momentum, bid/ask spreads) are computed strictly from past observations. The walk-forward splitter ensures calibration and test markets are never in the training fold.
- **No data corruption**: CSV fixtures are internally consistent; timestamps are monotonic.
- **No dependency issues**: All Python dependencies are installed and importable.
